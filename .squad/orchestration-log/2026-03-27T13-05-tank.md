## Key Decisions Made

1. **Test File Naming Convention**
   - `.osc` filename matches `.expected` filename (e.g., `fibonacci.osc` → `fibonacci.expected`)
   - Simple 1:1 mapping for test discovery and maintenance

2. **Negative Test Validation**
   - Test runner checks only that compiler returns non-zero exit code
   - Error message assertions deferred (can add later as spec matures)
   - Focuses on "does compiler reject invalid syntax?" not "exact error text"

3. **Expected Output Files**
   - Use exact byte-level comparison (no regex, no fuzzy matching)
   - Runtime output formatting must be deterministic
   - Ensures reproducible test results across platforms

4. **FFI Test Assumptions**
   - `c_abs` maps to C standard library `abs()` function
   - Compiler or linker must handle this name mapping
   - Other C FFI calls follow similar convention

5. **Build Artifacts Directory Convention**
   - Compiled test programs go in `tests/build/`
   - Directory is gitignored (`.gitkeep` ensures directory exists in repo)
   - Test runners create directory if missing on first run

### Test Infrastructure

**Bash Runner** (`tests/run.sh`)
- Compiles all `.osc` files to binaries
- Executes positive tests and compares output to `.expected`
- Verifies negative tests return non-zero exit code
- Cross-platform support (Linux, macOS)

**PowerShell Runner** (`tests/run.ps1`)
- Windows native test runner
- Same validation logic as bash runner
- Path handling for Windows filesystem
